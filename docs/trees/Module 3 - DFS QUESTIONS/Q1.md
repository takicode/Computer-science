## QUESTION 1: Search for a Value

This is the first real algorithm we'll build.

Not traversal for the sake of traversal.

Traversal with a purpose.

## Problem

Suppose someone asks:

Does this tree contain the value 7?

Tree:

          5
        /   \
       3     8
      / \   / \
     1  4  7  9

Should our program answer: true or false

depending on whether the value exists?

Yes.

That's today's goal.

This is one habit I want to build in you.

How would you look for 7?

Then the left subtree?
Then the right subtree?

We didn't invent a new traversal.

Step 2: The Recursive Thinking

Forget Java.

Imagine I ask you:

Can this tree find 7?

There are only four possibilities.

Case 1

The tree is empty.

null

Obviously:

false

Case 2

The root is 7.

Done.

true

Case 3

Maybe it's in the left subtree.

Ask the left subtree.

Case 4

Maybe it's in the right subtree.

Ask the right subtree.

That's the entire algorithm.

Notice how natural recursion feels?

Step 3: Turning Thoughts into Code

Let's translate each idea.

Case 1
if (node == null) {
return false;
}
Case 2
if (node.value == target) {
return true;
}
Case 3 & 4

Here's the interesting part.

Instead of writing another search algorithm...

we reuse the same one.

search(node.left, target)

and

search(node.right, target)
Complete Algorithm
public static boolean search(TreeNode node, int target) {

    if (node == null) {
        return false;
    }

    if (node.value == target) {
        return true;
    }

    return search(node.left, target)
            || search(node.right, target);
}

Don't memorize this.

Let's understand it.

Why ||?

Suppose:

Left subtree -> false

Right subtree -> true

Then:

false || true

equals:

true

Meaning:

"If either subtree finds the value, then the whole tree contains it."

Beautiful.

The Most Important Line
return search(node.left, target)
|| search(node.right, target);

This line is the heart of recursive problem-solving.

Notice what's happening.

We're not saying:

Search node 3.

We're saying:

Ask the entire left tree.

Exactly the recursive mindset we've been building.

Walkthrough

Search for 4.

          5
        /   \
       3     8
      / \   / \
     1   4  7  9

Call:

search(root, 4)

Execution:

5

Not found.

↓

Search left.

3

↓

Search left.

1

↓

Left = false

↓

Right = false

↓

Return false

Back to 3.

Search right.

4

Found.

Return true.

Now something interesting happens.

Because of ||, Java short-circuits.

Once the left side returns true, the right subtree of 5 is never searched.

That's an optimization built into the language.

## Engineering Corner

This is the first time we're writing an algorithm that can stop early.

Traversal always visits every node.

Search doesn't have to.

The moment the value is found, we're done.

This is a subtle but important distinction.

Complexity

Best case
Target is the root:O(1)


Worst case 
Target doesn't exist:Every node is visited: O(n)

where n is the number of nodes.

Notice this is not a Binary Search Tree.

We're searching an ordinary binary tree.

Without any ordering, we may need to inspect every node.

Interview Perspective

A common mistake is to write:

search(left);

search(right);

return false;

The problem?

You ignored the results of the recursive calls.

A recursive function should usually use the value returned by its recursive calls.

That's an important pattern you'll see over and over.

## Core Insight

Recursion is not the solution. Recursion is the way we repeatedly apply the same solution to smaller trees.

Before We End

Using a stack, we can implement the same DFS iteratively.

Later in our journey, we'll write both versions and compare them.

## The Pattern You Just Learned

I want to highlight something.

Today's lesson wasn't really about "search."

It was about a recursive pattern.

Look carefully:

if (base case)
return ...

return combine(
solve(left),
solve(right)
);

This pattern appears everywhere in tree algorithms.

For example:

Searching:

return left || right;

Counting:

return left + right + 1;

Finding maximum:

return Math.max(left, right);

Checking equality:

return left && right;

Height:

return 1 + Math.max(left, right);

Do you see it?

The recursive structure is almost always the same.

The only thing that changes is how we combine the answers from the left and right subtrees.

This is one of the most powerful ideas in recursive tree algorithms.

## Engineering Corner

I want to give you a habit that professional engineers develop.

Whenever you're solving a recursive tree problem, ask yourself these three questions:

1. What is my base case?

Usually:

if (node == null)

2. What question am I asking each subtree?

Examples:

Does it contain 7?
How many nodes are there?
What's the maximum value?
What's the height?

Notice you're asking the same question of both subtrees.

3. How do I combine their answers?

This is the heart of the algorithm.

Examples:

||
+
Math.max()
&&

If you can answer those three questions, you've already designed most recursive tree algorithms.

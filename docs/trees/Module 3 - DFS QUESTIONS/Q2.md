## Count the Number of Nodes
## The Problem

Given this tree:

          A
        /   \
      B       C
     / \     /
    D   E   F

Return:

6

Not print.

Not traverse.

Just return the total number of nodes.

Step 1: Solve It Like a Human

If I asked you:

"How many nodes are here?"

You'd probably count:

A = 1
B = 2
C = 3
D = 4
E = 5
F = 6

Easy.

But computers don't "see" the whole picture at once.

They solve problems one piece at a time.

Step 2: The Recursive Question

This is the most important question of the lesson.

Suppose I point at node B.

        B
       / \
      D   E

And I ask:

How many nodes are in THIS tree?

Not the whole tree.

Just B's tree.

The answer is:

3

Interesting...

Now I point at D.

D

Same question.

Answer?

1

Now E.

Answer?

1

Notice something?

We're asking the same question regardless of which node is the root.

That's recursion.

Step 3: Think Like the Root

Let's stand on node A.

          A
        /   \
      B       C

A doesn't know how many nodes exist.

But A does know something.

It has:

a left subtree
a right subtree

So A can ask them.

Left:
"How many nodes do you have?"

Right:
"How many nodes do you have?"

Suppose they answer:

Left = 3

Right = 2

Should A just return:

3 + 2

No.

Why not?

Because A forgot someone...

Who?

👉 Itself.

So the answer becomes:

1 + left + right

That 1 represents the current node.

This is the entire algorithm.

Step 4: Base Case

What if the tree is empty?

null

How many nodes?

Not one.

Not negative one.

The answer is:

0

So:

if (node == null) {
return 0;
}
Step 5: The Formula

We've just discovered it.

count(node)

=

1

+

count(left)

+

count(right)

Read it in English:

The number of nodes in a tree equals:

the current node,
plus every node in the left subtree,
plus every node in the right subtree.

Beautiful.

Step 6: Java Implementation

Don't copy it.

Read it.

public static int countNodes(TreeNode node) {

    if (node == null) {
        return 0;
    }

    return 1
            + countNodes(node.left)
            + countNodes(node.right);
}

Notice something?

Last lesson returned:

boolean

Today returns:

int

Same recursion.

Different information.

## The Big Insight

Notice something magical.

The recursive calls don't return:

true

or

false

They return information.

Information flows upward.

Like this:

       A
      / \
     /   \
    3     2

Each subtree solves its own problem.

The parent combines the answers.

Engineering Corner

This is called Divide and Conquer.

Not because we're splitting arrays.

Because we're splitting a problem into independent subproblems.

Every recursive tree algorithm does this.

Complexity

How many times is each node visited?

Exactly once.

Therefore:

Time:

O(n)

Space?

Recursion stack.

Worst case:

O(h)

where h is the height of the tree.

Notice...

Not O(n).

The call stack only stores the current path from the root to the current node.

That's why a balanced tree uses much less stack space than a skewed tree.

Compare with Search

Search:

return left || right;

Count:

return 1 + left + right;

Can you see the pattern?

Only the combination step changed.

Everything else stayed the same.

## Think Like an Engineer

Here's a question I ask myself whenever I solve a recursive tree problem:

What should each recursive call return to make my job easier?

For counting, it's an int.

For searching, it's a boolean.

For height, it's also an int.

For comparing trees, it's a boolean.

The return type isn't random.

It represents the information the parent needs from its children.

That's a subtle but powerful design principle.

## Core Insight

A recursive call is like asking a smaller tree to solve the same problem for itself. Your job is simply to combine the answers.

Before We End

Every lesson strengthens the same mental model:

Ask the right subtree.

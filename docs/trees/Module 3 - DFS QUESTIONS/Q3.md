## Count Leaf Nodes
## The Problem

Given this tree:

          A
        /   \
      B       C
     / \       \
    D   E       F
       /
      G

How many leaf nodes are there?

First...

Don't think about recursion.

Answer it manually.

## Step 1: What is a leaf?

Let's remind ourselves.

A leaf is a node that has:

left == null

AND

right == null

Not OR.

AND.

Examples:

D ✅

G ✅

F ✅

They have no children.

## Step 2: Solve It Like a Human

Looking at the tree:

          A
        /   \
      B       C
     / \       \
    D   E       F
       /
      G

Leaves:

D

G

F

Answer:

3

Easy.

Now let's teach the computer.

## Step 3: The Recursive Question

This is becoming a habit now.

Ask yourself:

What question should every subtree answer?

Answer:

How many leaf nodes are in your subtree?

Notice...

It's exactly the same style as:

How many nodes?

except now we're only counting leaves.

## Step 4: Base Case

Empty tree.

null

How many leaves?

Not one.

Zero.

if(node == null)
return 0;

Simple.

## Step 5: A New Situation

Suppose we're standing on:

D

What do we know?

left == null

right == null

That means...

D is a leaf.

How many leaves exist in D's subtree?

Exactly one.

So:

if(node.left == null && node.right == null)
return 1;

This is our second base case.

Notice this.

Today's algorithm has two base cases.

That's perfectly normal.

## Step 6: Internal Nodes

Suppose we're at:

      B
     / \
    D   E

Is B a leaf?

No.

So B shouldn't contribute:

1

What should B do?

Ask its children.

    Leaves in left subtree
    +
    Leaves in right subtree

That's all.

## Step 7: Recursive Formula

For an internal node:

        leafCount(node) = leafCount(left) + leafCount(right)

Notice something.

Unlike Count Nodes...

There is NO:

+1

Why?

Because internal nodes are not leaves.

That's today's key insight.

## Java Implementation

Try reading it like English.

public static int countLeaves(TreeNode node){

    if(node == null)
        return 0;

    if(node.left == null && node.right == null)
        return 1;

    return countLeaves(node.left)
         + countLeaves(node.right);
}

Read it:

If there's no tree...

Return 0.

If this node is a leaf...

Return 1.

Otherwise...

Ask the left subtree.

Ask the right subtree.

Add their answers.

Beautiful.

Compare with Count Nodes

Count Nodes

return 1 + left + right;

Every node contributes: 1

Count Leaves

return left + right;

Only leaves contribute:

1

Internal nodes contribute:

0

This is the first time we've seen conditional contribution.

## Engineering Corner

Imagine you're building a file explorer.

Folders are internal nodes.

Files are leaves.

Suppose someone asks:

"How many actual files are inside this directory?"

Would you count folders?

No.

Only files.

That's exactly what this algorithm does.

Complexity

Every node is visited exactly once.

Time:

O(n)

Recursion stack:

O(h)

where h is the height.

## A Pattern Is Emerging

Let's compare all our recursive formulas.

Search:

left || right

Count Nodes:

1 + left + right

Maximum Value:

max(node, left, right)

Count Leaves:

if leaf:
1

otherwise:
left + right

Different problems.

Same recursive structure.

## Think Like an Engineer

I want to introduce another design question.

When you're solving a recursive problem, ask:

Does the current node contribute anything?

Examples:

Count Nodes:

Always contributes 1.

Maximum Value:

Always participates in the comparison.

Leaf Count:

Only contributes if it's a leaf.

This question will become important when we solve path problems later.

## Core Insight

Recursive tree algorithms are often about deciding whether the current node contributes to the final answer, then combining the answers from the left and right subtrees.
